Proposal

Teaching Center Management System

AWS Serverless solution for learning center management project

1. Executive Summary

To address the challenges in operating an educational center, the Teaching Center Management System is designed to integrate all core business operations: Human Resource Management (HRM), Student Information System (SIS), Customer Relationship Management (CRM), Education Payment, and parent engagement. The goal is to optimize management processes, enhance operational efficiency, and improve the overall learning experience.

In the future, the system is oriented to expand with advanced technologies such as Artificial Intelligence (AI) and the Internet of Things (IoT) to personalize learning paths, develop virtual learning assistants, while modernizing infrastructure with smart classrooms and automated facility management.

2. Problem Statement

Current Issues

Educational centers today face difficulties in synchronizing management between online and offline classes. Processes such as course registration, attendance, progress tracking, or performance reporting are still mainly carried out manually or through multiple separate platforms, causing inconsistency and making data control difficult.

Additionally, there is a lack of a centralized system to store and analyze learning data in real-time. Relying on third-party solutions is not only costly but also complex, and difficult to customize according to specific needs. As the number of students and classes grows, the current system lacks scalability and automation, reducing operational efficiency as well as the learner experience.

Solution

The Teaching Center Management System is implemented based on an AWS Serverless architecture, optimizing costs, automatically scaling, and ensuring high availability.

Users access the system through Amazon Route 53, with requests passing through AWS WAF to filter attacks before distribution via CloudFront, accelerating page load and protecting against DDoS. Static web content is stored on S3 and quickly distributed via CloudFront.

User authentication is handled through Amazon Cognito, ensuring account security for students, instructors, parents, and administrators. After login, user requests are processed through Amazon API Gateway, triggering AWS Lambda to execute core business logic such as account management, class management, course registration, attendance, updating results, and schedule lookup. All data is stored in Amazon DynamoDB.

Documents, lecture videos, and assignments are stored in Amazon S3; system logs are sent to Amazon CloudWatch Logs for monitoring. CloudWatch Alarms and Amazon SNS automatically send alerts when errors or abnormal activities are detected. AWS Secrets Manager protects sensitive information such as API keys or service connection details.

The development and deployment process is automated through the AWS CodePipeline – CodeBuild – CloudFormation chain, ensuring seamless CI/CD, reducing configuration errors, and shortening deployment time. AWS CloudTrail and Amazon GuardDuty record API activity, detect threats, and protect the system from security risks.

The solution provides a unified, secure, scalable, and cost-optimized platform, helping educational centers manage both online and on-site training effectively, enhancing learning experience and instructor productivity.

Benefits and Return on Investment (ROI)

  • Accelerated development and deployment: Automated CI/CD (CodePipeline, CodeBuild, CloudFormation) reduces feature release time from several days to a few hours.
  • Operational cost optimization: Serverless architecture (Lambda, DynamoDB, API Gateway) charges only per request, saving 40–60% of costs compared to traditional EC2.
  • Comprehensive security: AWS WAF, GuardDuty, CloudTrail, and Secrets Manager enable early risk detection, filter web attacks, and protect sensitive data.
  • High access performance: CloudFront CDN combined with Route 53 accelerates access, reducing latency by 50–70%, while WAF protects the system from web attacks before content distribution.
  • Flexible scalability: The system automatically scales according to traffic without manual intervention.
  • Proactive monitoring: CloudWatch + SNS provides real-time alerts, helping the technical team respond promptly to incidents.

3. Solution Architecture

Detailed Description

  1. User Request Flow

    • Users open a browser and access the application domain.
    • Amazon Route 53 resolves the domain to an IP address and sends the request to AWS WAF.
    • AWS WAF performs:
      • Checks requests against configured rules (SQL Injection, XSS, IP blacklist, rate limiting…).
      • Blocks abnormal or malicious requests.
      • Forwards only valid requests to Amazon CloudFront.
    • CloudFront receives the request:
      • Checks the cache to return static content (HTML/CSS/JS) from S3 if available.
      • If content is not in cache, CloudFront fetches from S3 and returns it to the user with low latency.
    • Users receive the frontend and interact with the UI, generating API requests.

  2. Authentication & API Handling

    • When users log in:
      • Amazon Cognito authenticates login information (username/password or OAuth).
      • Cognito generates a JWT token and returns it to the client.
    • The frontend sends API requests with the JWT token to API Gateway.
    • API Gateway performs:
      • Verifies the JWT token with Cognito.
      • If the token is valid, the request is forwarded to the corresponding Lambda function.
      • If the token is invalid, API Gateway returns a 401 Unauthorized error.

  3. Lambda Processing & Data Access

    • AWS Lambda executes business logic:
      • Manages students, attendance, course registration, updates results, schedules, etc.
      • When accessing sensitive information (API key, DB password), Lambda calls AWS Secrets Manager.
    • Lambda reads/writes data to Amazon DynamoDB:
      • DynamoDB stores data in a NoSQL model, optimized for read/write, automatically scaling with traffic.
      • Supports queries using primary key (PK) or secondary index (GSI).
    • Lambda logs to CloudWatch Logs, including requests, errors, and execution metrics.

  4. Security & Threat Detection

    • AWS WAF protects the edge layer, filtering malicious web requests.
    • AWS GuardDuty:
      • Analyzes logs from CloudTrail, VPC Flow Logs, DNS Logs.
      • Detects abnormal behaviors (compromised IAM, unusual IP access).
      • Creates Findings and sends alerts via SNS or triggers automatic response Lambda when threats are detected.
    • AWS CloudTrail: records all API calls for audit and traceability.
    • Secrets Manager protects sensitive information and allows Lambda to access secret keys.

  5. Monitoring & Alerts

    • CloudWatch Logs collects logs from Lambda and API Gateway.
    • Metrics & Alarms:
      • Creates metrics from logs (CPU, error rate, latency, request count).
      • Configures CloudWatch Alarms to trigger alerts when thresholds are exceeded.
    • Amazon SNS sends real-time alerts to the operations team via email or HTTP/SMS endpoints.
    • Combines CloudTrail + CloudWatch for auditing API actions and overall security.

  6. CI/CD & Deployment

    • GitHub: stored in GitHub Repository.
    • AWS CodePipeline:
      • Receives webhook when code changes on GitHub.
      • Orchestrates the entire pipeline including stages: Source → Build → Deploy.
    • AWS CodeBuild:
      • Installs dependencies.
      • Runs unit tests.
      • Builds artifacts (ZIP, container image…).
    • AWS CloudFormation:
      • Deploys or updates the entire AWS infrastructure (API Gateway, Lambda, DynamoDB, S3, IAM Role).
      • Ensures infrastructure follows IaC model, consistent across Dev/Prod environments.
    • Automated CI/CD reduces errors and shortens deployment time.

  7. Summary

    • Request Path: User → Route53 → WAF → CloudFront → API Gateway → Lambda → DynamoDB → Lambda → API Gateway → CloudFront → User.
    • Security Path: WAF blocks web attacks → GuardDuty monitors logs → CloudTrail audits → SNS alert.
    • Monitoring: CloudWatch Logs & Metrics → Alarms → SNS.
    • CI/CD Path: GitHub → CodePipeline → CodeBuild → CloudFormation → AWS Resources.

4. Technical Implementation

Deployment Stages

  • Development Stage
    • Complete business logic and main flow for Lambda functions.
    • Write template.yaml file describing resources: API Gateway, Lambda Functions, DynamoDB, Cognito.
    • Use AWS SAM CLI to deploy code and template.yaml to LocalStack for local testing.
  • Deployment Stage:
    • Use AWS SAM CLI to deploy code and template.yaml to the real AWS environment.
    • Configure AWS CodePipeline and AWS CodeBuild to automate the CI/CD process.

Technical Requirements

  • Have an AWS account using Free Tier to deploy and use resources normally.
  • The template.yaml file must be correctly configured to fully describe all services.
  • The system must have an automatic rollback mechanism in case of deployment failure.

  • Internship (Week 1-12):

7. Risk Assessment

Risk Matrix

  • AWS configuration errors (IAM, Lambda, API Gateway, Cognito, WAF): High impact, medium probability
  • Exceeding AWS Free Tier limits: Medium impact, low probability
  • Data loss on S3/DynamoDB: High impact, low probability
  • Integration errors between AWS services: Medium impact, low probability
  • External attacks (DDoS, SQL injection, XSS): High impact, low to medium probability

Mitigation Strategies

  • AWS Configuration: Carefully check template.yaml, deploy on LocalStack before production deployment.
  • Free Tier Exceeding: Monitor costs regularly, set up Billing Alerts, optimize resources.
  • Data Loss: Enable S3 Versioning, periodically backup DynamoDB data.
  • Service Integration Errors: Ensure services operate in the same Region, verify IAM Roles and cross-service access permissions.
  • Application Security: Use WAF to filter malicious traffic, apply standard rules against OWASP Top 10, enable CloudFront and API Gateway logging.

Contingency Plan

  • Deployment failures: Rollback using AWS SAM CLI or restore previous Lambda version.
  • Budget overrun: Pause non-essential services, optimize architecture and resource usage.
  • Security incidents: Review CloudWatch/CloudTrail logs, update WAF rules, isolate affected endpoints.

  • Guarantee data safety with backup, versioning, and strict access control mechanisms.
